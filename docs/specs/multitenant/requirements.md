# マルチテナント化 — 要件

> 対象: 部活の練習計画ツール（Firebase Hosting + 単一 Cloud Function による SSR、名前付き Firestore、全データ Admin SDK 経由、エンジンは決定論で LLM 無し）。
> 現状は単一テナント（1校・男女2チーム固定・閲覧公開・編集はオーナー1名許可）。

## 背景（なぜやるか）

コーチ仲間に配布し、各自が自分のチームの練習計画を作れるようにする。
現状は1校分のデータを共有しており、別のコーチが自分のチームの計画を独立して持てない。

## 確定済みの方針（オーナー決定）

1. 閲覧もログイン必須。各コーチは自分のチームの計画だけを読み書きできる。他テナントの計画は閲覧もできない。
2. 登録は招待リンク制。プラットフォーム所有者が招待リンクを発行し、受け取った Google ユーザーが承諾すると、その人に新しい空テナントが払い出される。
3. ドリルカタログ（216件）は全テナント共有。

## スコープ内（このリリースでやること）

- データをテナント単位に分離する（テナント＝コーチの作業場。配下に男女のチームを内包）。
- サインアップ（招待リンクの承諾でテナント払い出し）。
- 閲覧・書き込みを、所属テナントの所有者にスコープする。
- 描画・編集・エンジンをテナント識別子でパラメタ化する（エンジンと描画ロジックは無改変、データ境界だけ変更）。
- 既存の現行校データを最初のテナントへ移行する。

## スコープ外（今回やらない・将来）

- 1テナントに複数コーチを同居させる「参加」型の招待（権限昇格リスクのため新規払い出しと物理分離して後送り）。
- 男女2チーム以外の構成（単独チーム校・3チーム以上）。現行の2チーム前提は構造として残す。
- ドリルのテナント別追加（共有カタログのみ）。
- メール送信による招待配送（リンク文字列を所有者が手渡す前提。配送は将来）。

## 受け入れ条件（行動で確認できる形）

- 未ログインで本番URLを開くとログイン画面に誘導され、計画は見えない。
- 所有者がログインすると、自分のテナントの計画だけが表示・編集できる。
- 別テナントの識別子を指定しても、所属していなければ閲覧も書き込みも拒否される（越境不可）。
- 招待リンクを承諾した新規コーチに空テナントが払い出され、エンジンの叩き台がすぐ表示される。
- 既存の現行校データが最初のテナントへ移行され、所有者ログインで従来どおり表示・編集できる。
- ローカル（エミュレータ）は従来どおり無認証で書き込めて、複数テナントの分離を E2E で検証できる。
