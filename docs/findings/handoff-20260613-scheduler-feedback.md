## 実装した機能（差し戻し4系統＋レビュー対応）

### A. タイポ階層の再設計（6段→5段・見出し>本文）

- **問題**: 旧スケール（22/17/15/14/12/10）は見出しラベル（「ねらい」「やり方の要点」「メタ」「本日の狙い」「目標」「チェックする数字」等）を 10px に置き、**見出しが本文（14px）より小さく**、どれが見出しか判別できない構造化文書違反だった。
- **対応**: 新スケール **27 / 22 / 17 / 14 / 12**（約1.25比・5段）。H1=27（ページタイトル）/ H2=22（日ヘッダ・セクション見出し・詳細のドリル名）/ H3=17（フィールド見出し・列見出し・KPI値）/ 本文=14（カード見出しは同サイズ太字700で差を付ける）/ 補助=12（アイブロウ・時刻・分数・チップ・脚注）。見出し役は必ず本文よりサイズ上位。`firebase-kit/.agents/skills/hallmark/references/typography.md`（1ページ5段まで・差はサイズと太さと色で）準拠。
- **ファイル**: `ui/render-shared.mjs`（共通CSS・T6コメント）、`ui/pattern-timeline.mjs`（パターンCSS・H1インライン）

### B. 装飾整理（枠線の意味論）

- **原則を新設**: 枠線（border）は「押せるもの（ボタン・タブ）」と「カード面の区切り」だけに使い、**静的メタ情報をピル化しない**。
- 時刻表示 `.tk`（日タイムライン・中央スパイン）の枠線ピル撤去（背景は中央線マスク用に維持）／「両チーム共通」バッジ（`.bothmark`）を**削除**（左右ミラー帯＋ページ冒頭の説明で自明・ごちゃつきの主因）／日ヘッダのコート・時間メタ、月カードのフェーズ表示を脱ピル化。
- **ファイル**: `ui/render-shared.mjs`、`ui/pattern-timeline.mjs`

### C. 詳細画面「← 戻る」の不具合修正

- **問題**: 旧実装は `overlay.querySelector('.drill-close')` で**先頭1パネルのボタンにしかリスナーが付かず**、2枚目以降のパネル（全16可視トリガー・カタログ215件）で戻るが無反応だった。
- **対応**: オーバーレイへのイベント委譲（`e.target.closest('.drill-close')`）に変更。`file://` 環境で `history.pushState` が throw する場合の `try/catch` フォールバックも追加。
- **ファイル**: `ui/render-shared.mjs`（`clientScript`）
- **検証**: Playwright で別パネル2件の開閉を実機検証（`closedHidden:true`）。検証スクリプトは恒久リポジトリに残さず追跡から除外（後述F）。

### D. 整理運動（ダウン）の二重表示解消＋内容の整流

- **問題1（二重表示）**: タイムライン末尾に UI 固定で描画していた「ダウン／ミーティング」帯が、エンジンが生成する CD（クールダウン）ブロックと二重に出ていた。→ 末尾は「終了」マーカー（振り返りひとことで解散）に変更し、整理運動本体はエンジンの CD ブロックが担う。
- **問題2（内容が実態と違う）**: オーナー指摘「ダウンは体育館を軽く2周してスタティックストレッチでは」。カタログに **CND-029「クールダウンジョグ（体育館2周）」**（duration 2-3分・低強度・philosophy_tags=クールダウン/整理運動/リカバリ）を追加。CD ブロックの並びを `coolDownStage()`（0=軽有酸素ジョグ/ウォーク → 1=静的ストレッチ → 2=呼吸・リカバリ）で整流し、「ジョグ→静的ストレッチ」の順に出るようにした。
- **複数itemのCDバンドル可視化**: `_bundleMainItem()` が先頭以外を `components` に畳み込み、ジョグと静的ストレッチの両方をタイムライン帯（`tbl-comp`）に表示。
- **ファイル**: `ui/pattern-timeline.mjs`（終了マーカー）、`ui/render-shared.mjs`（配布テキストの締め）、`engine/src/allocate.js`（`coolDownStage` 追加・cdPool ソート）、`docs/practice-knowledge/data/drills.json`（CND-029 追加）、`engine/test/coaching-mode.test.js`・`engine/test/needs-coach.test.js`（カタログ件数 214→215）、`ui/rotation.mjs`（`_bundleMainItem`）

### E. 5分刻み化

- **問題**: オーナー指摘「全メニュー5分刻みのはず、なぜ12分が出る」。組違いの前後半分割が `Math.floor(block.minutes/2)` で、25分→12/13 の端数を生んでいた。
- **対応**: `half1 = Math.floor(block.minutes/10)*5`（25分→10/15）。配布テキストの束ね（WU/CD）内訳の端数分数（1分・2分等）は非表示にし、時間はブロック単位の5分刻みに一本化。
- **ファイル**: `ui/rotation.mjs`、`ui/render-shared.mjs`（`plainText`）

### F. push 前コードレビュー指摘の反映（7観点レビュー）

- **採用1（退行バグ・本質）**: E の5分グリッド化に、`block.minutes % 5 !== 0 || block.minutes < 10` で **throw** する防御が混ざっていた。`clamp(round5(min(owedMin,20)),5,owedMin)` は `clamp(5,5,3)=3`（clamp は `min(hi,max(lo,x))`）で端数の philosophy-floor ブロック（3分・5分）が数学的に生成可能。throw は以前クラッシュしなかったプランを落とす退行。→ **5分で割れる時だけ前後半分割し、割れない/10分未満は分割せず1行（コーチ側のみ・実尺占有）で出すフォールバック**に修正。回帰ユニットテスト（8分ブロックで throw せず1行・時計不変）を `ui/rotation.test.js` に追加。
- **採用2**: 一時検証スクリプト `ui/e2e-verify-detail.mjs` を追跡から除外（CI 非接続・腐敗リスク）。
- **対応不要（理由明示）**: ①`coolDownStage` の文字列マッチが `COOLDOWN_SUBSKILL_RE` と語彙重複・将来はカタログに `cooldown_stage` フィールドを持たせるのが根本（設計正本§8の notes構造分離と同根・別タスク）②`components` 生成が plan-data（WU）と rotation（CD）の2系統（統合は別タスク）③cdPool ソートの comparator が `coolDownStage` を毎回再評価（プール数件で無視可）④jog(2分)先頭詰めで静的ストレッチ押し出し（実カタログは jog2+静的3=5分で収まる・非空CD保証あり）⑤half1反転（25→10/15で前半が短い・組違いで前後半入れ替わるため公平性に影響なし）。
- **ファイル**: `ui/rotation.mjs`、`ui/rotation.test.js`

## 開発中に得た教訓

### 1. デザイン規範を「禁止事項の検査」だけに使い「あるべき姿の設計」に適用しなかった

**何が起きたか**: hallmark typography 正本（1ページ5段まで・見出しは本文より上位）を持ちながら、anti-pattern grep（border-left・汎用書体・gradient 等の検出ゼロ確認）にしか使わず、見出しが本文より小さい構造化文書違反を作った。オーナーから「構造化文書を正しく意識しろ」「デザイン例を持っていないのか」と差し戻し。

**再発防止**: デザイン規範は「やってはいけないこと（anti-pattern）」と「やるべきこと（見出し>本文・スケール段数・階層）」の両面を適用する。HTML/CSS 生成時は anti-pattern grep に加え、見出しと本文のサイズ関係・段数を実出力で確認する。

**反映先候補**: wasurenagusa（保存済み）

### 2. 配布フォーマットの粒度要件を分割ロジックに組み込まず機械的半分割で実装した

**何が起きたか**: 「全メニュー5分刻み」要件があるのに、組違いの前後半を `floor(minutes/2)` で機械的に割り、25分→12/13 の端数を生んだ。

**再発防止**: 粒度要件（5分刻み等）は分割・配分ロジック自体に組み込む（`floor(minutes/10)*5`）。出力の単位がフォーマット要件に乗っているかを実出力で確認する。

**反映先候補**: wasurenagusa（保存済み）

### 3. 防御的 throw が「以前動いていたケース」を落とす退行を生んだ

**何が起きたか**: 5分グリッド非適合ブロックを throw で検出する防御を入れたが、端数の philosophy-floor ブロックは `clamp(5,5,3)=3` で実際に生成可能で、以前クラッシュしなかったプランを落とす退行になった。コードレビューで finder の見解が割れ（到達可能 vs 到達不能）、`clamp` 実装を読んで到達可能と確定。

**再発防止**: 「壊れたら気づく」防御 throw を入れる前に、その入力が上流（clamp/round/分配ロジック）で実際に到達可能かを実装で確認する。到達可能なら throw でなくフォールバック（degrade）にする。レビューで finder の見解が割れたら多数決でなく該当実装を読んで決着させる。

**反映先候補**: wasurenagusa（保存済み）
