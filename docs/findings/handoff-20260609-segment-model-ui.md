# 引継資料: 練習計画エンジンの段モデル化とUI3見せ方（2026-06-09）

## 概要
### このセッションで達成したこと
- 練習計画エンジンの時間配分を「固定％ブロックを多数の短いドリルで埋める」方式から、**コーチの実メニューに倣った「少数の持続する段（セグメント）」方式**へ作り直した。1日の練習が、テーマ単位の持続塊（対人20分・技術30分・試合25分…）で出るようになり、ゲームに個人ドリル（マイカン/ツーメン）が混入する事故が消えた。
- 練習計画UIを、配布できるドキュメント型で**3つの見せ方（配布プリント／タイムライン／週間ボード）×4レベル（年/月/週/日）**として実データから生成するように作り直した。印刷PDF・貼れるテキスト・組違いローテ2レーンを各案が備える。
- オーナーは**タイムライン案（`pattern-timeline.mjs`）を採用**と表明。

### なぜやったか
前セッションのUI試作が「細切れ生成・5分刻み未満・時刻4桁無コロン・カテゴリ不適合ドリル混入」で却下された。さらにオーナーから本セッションで「参加人数（男女各10名前後）を踏まえた適正な時間配分から始めろ」「スイッチコストを考えろ（1h=2〜3段／2h=3〜5段／3h=4〜7段が限度）」「実メニューの型に合わせろ」という設計指示が追加され、時間配分モデルそのものの作り直しが必要になった。

## 実装した機能

### エンジン: 段（セグメント）モデル
**主要ファイル**

| ファイル | 役割 |
|---|---|
| `engine/src/allocate.js` | 全面書き換え。1日を WU＋数段の主テーマ＋CD で組む。`computeSessionShape(分)` が WU/CD/主予算/段数帯/目標段数を返す。`chooseMainSegments()` が段のカテゴリと尺を決め、`pickSegmentDrill()` が各段の主ドリル＋「いずれか」候補を選ぶ。 |
| `engine/src/format.js` | 各段の「いずれか」候補を表示する行を追加。 |
| `engine/src/types.js` | `PlanItem` に `alternatives:[{drill_id,name}]`（「いずれか」候補・表示のみ）を追加。 |
| `engine/test/allocate.test.js` | 段モデル用に全面更新（段数帯・カテゴリ適合・持続15分以上・5分丸めを業務意図で検証）。 |
| `engine/DESIGN.md` | §4 を段モデルの仕様に書き換え。 |

**設計の要点**
- **スイッチコスト境界**: 主段（WU/CDを除くカリキュラム段）の数を尺で上限する。帯は `[round(分/60)+1, round(分/30)+1]`（60/120/180分でちょうど 2-3 / 3-5 / 4-7）。目標は `round(主予算/27)` を帯でクランプし、少なく長い側へ寄せる。
- **カテゴリ適合**（前回却下理由の根本）: 各カテゴリを `CATEGORY_BLOCK` で 技術/対人/ゲーム のいずれかに限定。技術=個人技術（シュート/フィニッシュ/ハンドリング/パス/フットワーク）、対人=対人・チーム戦術（1on1/チームDF/チームOF/リバウンド）、ゲーム=意思決定/ゲーム形式。フィニッシュ系ソロが試合ブロックに入らない。
- **持続段**: 各主段の尺は割り当て分で走る（≥15分・5分丸め・上限40分）。1段＝主ドリル1本＋同カテゴリの「いずれか」候補（最大2本）。ドリルの自然長とは切り離す。
- **試合の締め**（コーチの定石）: 重みだけでは上位に来ない `意思決定/ゲーム形式` を1段予約し、各コーチ日が試合形式で締まる。
- **守備/速攻フロア**: 在席日に守備（オールコートマンツー）と速攻（アーリー/トランジション）を週内最低40分ずつ確保（`floorTracker`）。コーチ不在日は対人を置かない（自走不可なので正しい挙動）。
- **WU/CD は従来どおり**: 多数の短いストレッチの束（スイッチコスト対象外）。CDはWUと同一ドリルを避けつつ空なら重複許容で非空保証。

**動作確認**: `engine/data/config.sample.json`（火金=在席100分/水木=不在100分/土=在席165分）で `node engine/bin/plan.js` を実行。火＝WU15→技術35→対人20→試合25→ダウン、土＝WU25→技術70→対人40→試合20→ダウン。段数帯・カテゴリ適合・対人15分以上・5分丸め・試合の締め・フロア40分を機械検査でゼロ違反確認済み。

### UI: 配布ドキュメント型 ×3見せ方 ×4レベル
**主要ファイル**

| ファイル | 役割 |

**注**: 生成HTML（`ui/*.html`）は `.gitignore` 済み（再生成: `node ui/build.mjs`）。ソースは `.mjs` のみ追跡。
**配布**: 印刷/PDF・テキストコピー・年/月/週/日の4レベルタブ（クリックで切替＝動作確認済み）・組違いローテ2レーン横。

**Hallmark監査**: 全3案を実体grep（gradient/側帯/汎用書体/transition:all/100vh/gradient見出し/emoji/eyebrow＝全て0）＋全レベル目視で確認。タイムラインに `::before` の左色帯（grep回避だが目視で側帯）を1件発見し、ブロック色をレールのドット＋段ラベル色へ移して除去済み。

## 既知の注意点・未対応項目（優先度順）
1. **【最優先・オーナー指摘】年/月レベルのシーズン構造が誤り**。`ui/plan-data.mjs` の `buildYearBands()`／`buildMonthWeeks()` は単線フェーズ帯で、中学チームの実構造を表していない。正しくは:
   - 学校年度は**4月始まり**（7月は起点ではない）。
   - シーズンは**2チームの重なり**: 「現チーム（〜夏の大会で引退）」と「新チーム（夏の代替わりで始動〜翌夏）」。
   - 夏の大会（6〜7月: 中野区選手権→東京都選手権）= **現チームの終わり**（現3年最後の公式戦）。直後の代替わりで新チーム始動。
   - **2つのピーク**: 夏の大会（現チーム）＋ 新人大会（新チーム: 区新人11〜12月→都新人1〜2月→3月研修・1年生大会）。
   - **男女で時期がズレる**: 男子は都選手権（7月下旬）まで長く新チーム始動が後ろ倒し、女子は早く代替わり。男女2チームの山が別々（体育館は共有）。
   - 出典: `docs/practice-knowledge/reference/チーム基本情報.md` の「年間大会カレンダー」「シーズン構造と代替わり」「令和8年度の見込み」節。
2. エンジンの `config.phase`／`current_month` は単一の系列。上記の現/新チーム・男女を意識した phase 導出になっていない（年/月UIと連動して見直す）。
3. WU/CDの個別ドリルが1分刻みで表示される（実メニューも同様だが、束としての見せ方は磨き余地あり）。
4. CDが尺に対し短くなる日がある（カタログに静的整理運動が少ないデータ要因・非空は保証）。
5. 実選手データ（Drive「バスケ_選手名簿」2026-06-01更新）・男女コート割りは未接続（サンプルKPIのまま）。実名は機密（PII）なのでUIはサンプルで作成済み、接続はオーナー受領後。

## 開発中に得た教訓
1. **年間/シーズン構造はドメイン原典を読んでから設計する**。何が起きたか=年フェーズ帯を `current_month` と一般論だけで仮置きし、4月始まり・現/新チームの2サイクル・男女別の代替わりという中学部活の実構造を無視して「7月起点」に見える誤りを出し、オーナーに叱責された。再発防止=年/月/シーズンを扱う前に `docs/practice-knowledge/reference/チーム基本情報.md` の「シーズン構造と代替わり」を必ず読み、2チーム重なり・2ピーク・男女ズレを一次情報として反映する。反映先=Wasurenagusa（プロジェクト固有）。
2. **サブエージェントのHallmark自己申告は目視で裏取りする**。何が起きたか=並列生成エージェントが「アンチパターンgrep全0」と報告したが、タイムライン案は `::before` で左色帯（側帯）を作りgrepを回避していた。実体は目視で一発アウト（オーナーが最も嫌う「Divの左に線＝AI」）。再発防止=デザイン成果はgrepだけでなく必ずheadless Chromeでスクショして目視。grep回避の擬似要素側帯まで疑う。反映先=Wasurenagusa＋既存ルール（CLAUDE.md デザイン領域の絶対原則 layer2）の運用で対応済み。
3. **「複数パターン」は別ファイルの別デザインで、レベルタブと混同させない**。何が起きたか=「日/週/月/年」のレベルタブと「3パターン」をオーナーが取り違え「どこが3パターン？」となった。再発防止=パターン（見せ方）とレベル（時間粒度）を提示時に明確に分け、index.html（選択トップ）を主導線にして3案を別タブで並べて見せる。反映先=Wasurenagusa。
