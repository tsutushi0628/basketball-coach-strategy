- **step-by-step の番号付き手順書**は存在しない。`notes` は自由文1〜数文。番号付きステップ表示は作らない（捏造になる）。

### 1.4 notes の配布用クレンジング（PdM裁定2・決定論・LLM不使用）

notes には **138/214件**にAI調査メモ（出典補完・検証記録）が混在する（実データ走査で確定）。配布画面に出すと「AIの調査メモ感」がオーナー酷評の温床になるため、決定論ルールで落とす。

**ルール（純関数 `cleanDrillNotes(notes: string): string`・PdM条件①②準拠）**:
1. `notes` を句点（`。`）で文に分割する。
2. 次の固定語を**含む文を丸ごと落とす**: `要出典補完`／`出典`／`fetch`／`WebFetch`／`403`／`404`／`検索`／`サイト本文`／`ページ正式`／`指標化候補`／`別エントリ参照`／`bot`／`URL`／`動画あり`（動画有無は `video_url` のリンク表示が担うため重複メタ）。
3. 残った文を `。` で再結合して「やり方の要点」本文にする。
4. **元データ（drills.json）は書き換えない**。registry 生成時の表示層適用のみ（PdM条件②）。
5. ユニットテストを付ける（PdM条件①）: 代表入出力（メモ混在→やり方のみ残る／全文メモ→空文字／メモ無し→原文不変）を業務期待で検証。

**実測結果（このルールでの全件走査・仕様の根拠）**:
- 部分削除: 138件。クレンジング後に**空になるのは10件**（全件 notes が「要出典補完」のみのドリル: HND-004・SHT-017・SHT-020・FIN-011・PAS-003 等）。
- **空化10件のフォールバック（PdM条件③）**: 「やり方の要点」見出しの下に**「このドリルの手順はカタログに未記載（ねらいとメタ情報を参照）」と1行明示**する（沈黙フォールバック禁止。見出しごと消すと「データがあるのに出していない」のか「無い」のか読み手が判別できない）。
- **既知のトレードオフ（安全側を選択）**: 一部ドリル（FIN-002〜007等）は実際の手順説明が調査メモ文の括弧内に混在しており、文ごと落とすと要点が薄くなる。括弧内の救出抽出は決定論ではメモ露出ゼロを保証できないため行わない。**配布物へのメモ露出1件は要点の薄さより致命的**（オーナー酷評の再発点）という優先順位判断。恒久対策はカタログ側の構造分離（§8自発添え）。
- **完了条件（PdM条件④）**: 生成HTMLの詳細部分を `要出典補完|WebFetch|403|bot遮断|検索で|出典ページ|URL訂正|URL補正` で grep して**検出ゼロ**。QAゲートに含める。

## 2. 画面構成の決定（メニュー詳細画面）

### 2.1 現状

- rotation 日（火金）は中央スパインの下に `<details class="menu-detail">` の折りたたみ「共通メニュー（男女同一の内容）」があり、開くと同じ内容の1列タイムラインがもう一度出る（`ui/pattern-timeline.mjs` `dayTimeline()`）。
- ドリル名はただのテキスト。実施方法に当たる情報はどこにも無い。

### 2.2 決定: 遷移方式 = ページ内オーバーレイ詳細（PdM裁定1・条件込みで「画面」として成立させる）

**PdM条件「モバイルでは全画面の画面として振る舞う・戻る1タップ」の専門観点点検: 以下の4条件を満たせば「画面」として成立すると判定（満たせるので採用）**:

| 「画面」要件 | 設計での充足 |
|---|---|
| 全画面占有（小モーダル禁止） | 狭幅（≤680px）では `position:fixed; inset:0` の**全画面シート**（幅100%・高さ100dvh・縦スクロール可）。広幅では中央パネル（max-width 560px）＋暗幕 |
| 戻る操作が1タップで明確 | シート上端に**固定ヘッダ「← 戻る」**（タップ目標44px・スクロールしても常時可視）。広幅は同位置の「✕ 閉じる」＋暗幕タップ＋Esc |
| OSの「戻る」操作で閉じる | **URLハッシュ駆動**で開閉する: 開く=`location.hash = 'drill-<id>'`、閉じる=ハッシュ除去。`hashchange` 監視で表示を同期するため、**Android/iOSの戻るジェスチャ・ブラウザ戻るボタンが「詳細を閉じる」として自然に機能**する（履歴に1段積まれる）。タブ切替と同型の hidden トグル＋ハッシュ同期で、追加スクリプトは20行規模 |
| 画面としての状態保持 | ハッシュがそのまま詳細画面のURL状態になる（リロードしても同じ詳細が開く）。閉じると元のタイムラインのスクロール位置に戻る（オーバーレイは背面を動かさない） |

**現状→決定→理由**:
- **決定**: タイムライン上のドリル名（日スパイン・1列タイムライン・「いずれか」候補名）を `data-drill` 付きのタップ可能要素にし、タップでそのドリルの詳細オーバーレイを開く。折りたたみ「共通メニュー」は廃止（`<details>` と `.menu-detail` 系CSS・print規則を削除。rotation日は中央スパインのみ、非rotation日は `menuTimeline` を直接展開）。
- **理由（行動観点）**: コーチは体育館で片手スマホ・タイムラインを上から追っている。ドリル名タップ→やり方確認→戻る、の往復が最小コスト（開く1タップ・戻る1タップ・**元のスクロール位置が保存される**）。当初案のページ内アンカー（詳細セクションへジャンプ）は印刷親和だが、戻りでスクロール文脈を失い、日ビュー下に40件超の常駐セクションができて「日タブ=1画面」の密度を壊すため、オーバーレイに置き換える（§7）。
- **「いずれか」候補の連鎖**: 詳細内の代替候補名もタップ可能。ハッシュが差し替わり同じ機構でパネルが切り替わる（戻る操作で1つ前の詳細に戻れる=履歴が自然に機能）。

### 2.5 印刷時の詳細の扱い（PdM条件・確定）

**印刷に詳細は含めない（確定）**。オーバーレイは `@media print { display:none }`。
理由: 配布印刷物の主用途は「当日の段取り1枚」（実物Doc型・時間ブロックの紙）であり、40件超の詳細カードを付録すると紙束化して配布性を壊す。詳細は画面（スマホ）で見る前提に役割分担する。将来「付録つき印刷」の需要が出たら、生成済みパネルHTMLを print 専用に通常フロー展開するだけで実現できる拡張点を残す（別タスク・§8）。

### 2.6 実装方式・実装箇所

- **詳細registry**: 新ファイル `ui/drill-detail.mjs` — 素の drills.json を読み、名前キーで `{id, name, subSkill, aim, metricMeaning, notesClean, court, balls, people, durationMin, durationMax, grades, intensity, loadNotes, masteryStage, video, sourceName, sourceUrl}` を返す `buildDrillRegistry()` と、純関数 `cleanDrillNotes()` を export。AIM_MAP/SHORT_CAT は `plan-data.mjs` から import（重複定義しない）。
- **パネル生成**: `ui/pattern-timeline.mjs` 新関数 `drillDetailPanels(data, registry)` — 当週計画に登場する全ドリル（主＋いずれか候補＋rotation補充。実測40件規模）のパネルHTMLを全件 hidden で生成し、日レベル末尾に置く。
- **タップ要素化**: `rotationTimeline()`・`trackSeg()` のドリル名と `altLine()`（`render-shared.mjs`・registry を引数で受ける形に変更）を `data-drill="<id>"` 要素に。
- **開閉スクリプト**: `render-shared.mjs` `clientScript()` に追記 — `[data-drill]` クリック→`location.hash` 設定、`hashchange` でパネル表示同期、戻るヘッダ/暗幕/Escでハッシュ除去（20行規模・既存タブ切替と同型）。
- **新規トークン**: `--scrim: rgba(42,32,26,.32)`（暗幕・`--ink` 系半透明。既存に幕用半透明値が無いため。追加はこの1値のみ）。

## 3. タイポグラフィスケール表（6段・全量割当）

### 3.1 現状

フォントサイズが 9/10/11/12/13/14/15/17/21/26px と10種以上散在し、段の意味（見出し階層）と結びついていない。「コーチ付き」タグ（11px・橙塗りピル）がドリル名（14px）より視覚的に強く、カード内が「タグ行／名前行／分数行」の3行でスペースを浪費している。

### 3.2 決定: 6段スケール（全段整数px・約1.2倍刻み）

ドラフトの 26/20/16/14/12/**10.5** 案は不採用（小数pxはレンダリング揺れを生み「規則的」の趣旨に反する。26/20はフラット化後の紙面で過大）。clamp も全廃して固定値にする（規則性優先）。

| 段 | px / weight | 役割 | 割り当てる全要素（クラス名） |
|---|---|---|---|
| **H1** 22px/700 | ページ題 | `h1`（school名＋練習タイムライン） |
| **H2** 17px/700 | レベル見出し・日見出し・詳細画面のドリル名 | `.lvh`、`.dayhead .dh-t`、詳細パネル見出し |
| **H3** 15px/700 | カード見出し | `.monthcard .mc-mon`、`.kteam .kth`、`.wg-dayhead .wd`（曜日） |
| **本文** 14px/600（地の文400） | 主役テキスト | `.spine-name`・`.tdn`（ドリル名）、`.spine-band .tbl`、`.dh-aim`・`.mc-aim`・`.gline .txt`、`.ev .evn`、`.daytab`・`.lvtab`・`.btn`、詳細の「ねらい」「やり方の要点」本文 |
| **補助** 12px/400 | 二次情報 | 分数（`.spine-mins`・`.tlm`・`.tbm`・`.ev .evm`）、`.alt`、`.tk`・`.wg-tick`（時刻）、`.note`・`.assume`・`.inote`、`.kpi .name/.val`、`.arccell .am`、`.vid`、詳細のメタ行・注意・参考元、`.wg-dayhead .wt`、`.ev .evt`、ページ説明文 |
| **ラベル** 10px/700・letter-spacing .06em | 種別・状態ラベル | `.tag`（→§3.3で形も変更）、`.dh-aiml`、`.gline .lab`、`.mc-phase`、`.peakchip`・`.nowchip`、`.bothmark`、`.spine-half`、`.tc-half`、`.wg-dayhead .wr`、`.ev .evl`、`.tbl-block`、`.arccell .ap`、`.foot`、凡例、詳細のセクションラベル |

- 9px・10.5px・11px・13px・15px(非見出し)・21px・26px・clamp は全廃し、6値（22/17/15/14/12/10）に snap。9px（`.wg-dayhead .wr` 等）の下限割れは10pxへ引き上げ（可読性下限）。
- `tabular-nums`（時刻・分数・KPI）は既存どおり全段維持。
- 重要度とサイズの対応原則: **ドリル名（14px）＞分数・時刻（12px）＞状態ラベル（10px）**。タグが名前より目立つ現状の逆転を正す。

### 3.3 カード内の凝縮（「コーチ付き／メニュー名／分数」3行 → 1〜2行）

**決定: 案A（1行基本＋条件付き2行目）を採用。**

```
[現状・3行+α]                     [決定・案A]
┌ コーチ付き(橙ピル11px) ┐        ┌ ● 11メン速攻（…）動画  12分 コーチ ┐ ← 1行目
│ 11メン速攻（…）動画    │        │ いずれか ドラッグスクリーン ／ …    │ ← altがある時だけ
│ 12分                   │        └──────────────────────┘
│ いずれか …             │
└────────────────┘
```

- 1行目: **モードdot**（コーチ付き=6pxの `--orange` 塗り点・行頭。ドラフト案を採用——グレースケール印刷や色弱でもラベルと冗長化でき、最小占有で視線アンカーになる）＋ドリル名（14px/600・タップ要素）＋分数（12px mute・`margin-left:auto` 右寄せ）＋モードラベル（10px/700・**塗りピル廃止→文字のみ**。コーチ付き=`--orange-deep`「コーチ」・レクチャ=`--terra`「レクチャ」）。
- **自走はdot・ラベルとも非表示**（ドラフトは自走にも○dot＋「自走」を置くが不採用——自走が多数派の画面で全行にマークが付くと「マークが付いている」の希少性が消え、コーチ付きの判別力が落ちる。自走の判別は§4の面の濃淡＋凡例が担う）。例外: コーチ不在日（水木・全段自走）は面差が出ないため日ヘッダの「コーチ不在」表記（既存）で日単位に伝える。
- 長いドリル名は1行目内で折り返しを許す（ellipsisで切らない。配布物なので名前の完全性優先）。
- `modeTag()` は dot＋文字ラベルの `modeMark()` に置換（`render-shared.mjs`。自走時は空文字を返す）。
- 適用箇所: `.spine-side`・`.tdrill`・`.tccell`・詳細パネルのモード表示。
- **週グリッド mini イベントの下限是正**（ドラフト§3.4採用）: `.ev.mini` の高さ下限を現行 `Math.max(10,…)` から**14px**（10pxラベルが収まる高さ）へ。土曜末尾の極小ブロックが潰れて読めない実害の是正で、タイポ規則化（最小段を割らない）の一部。

## 4. フラット化の具体（シャドウ廃止・印刷適性）

### 4.1 現状

ニューモーフィズム3トークン（`--shadow` 大影・`--shadow-soft` 小影・`--inset` 凹み影）を生成HTML中42箇所で使用。画面は影で面を浮かせ、印刷時だけ border に差し替える二重定義（print 規則）。影は区切りの判別性が低く、印刷でグレーの滲みになる（オーナー指摘どおり）。

### 4.2 決定: シャドウ全廃 → 「1px罫線＋面の濃淡」の2手段だけで区切る

| 廃止するもの | 代替（決定） |
|---|---|
| `--shadow`（`.dayhead`・`.weekgrid`・`.lvtab.on`・`.daytab.on` 等） | `background:var(--surface)` ＋ `border:1px solid var(--line-2)`（濃罫線22%。選択中タブは `--orange` 塗り維持で罫線不要） |
| `--shadow-soft`（カード類 `.tlcard`・`.spine-coach`・`.ev.coach`・`.monthcard`・`.goals`・`.arccell`・`.note`・`.btn` 等） | `background:var(--surface)` ＋ `border:1px solid var(--hair)`（淡罫線9%） |
| `--inset`（`.tk`・`.dh-court`・`.dh-aim`・`.tag-self`・`.vid`・`.spine-self`・`.ev`・`.kteam`・`.modetoggle`・`.arccell.peak1` 等） | `background:var(--bg)` ＋ `border:1px solid var(--hair)`（沈んだ面=地色、浮いた面=surface の**面の濃淡2値**で凹凸の意味を代替） |
| hover の `transform:translateY(-2px)` | 維持（影に依存しない控えめなフィードバック） |
| print 規則の `box-shadow:none;border:…` 差し替え | **削除**（画面が最初から罫線になり二重定義が消える。画面=印刷の WYSIWYG） |

- トークン定義 `--shadow`/`--shadow-soft`/`--inset` は `TOKENS` から**削除**（残すと次の編集者が再使用して漂流する）。完全な影ゼロ（生成HTML grep `box-shadow` = 0件）をゲートにする。ドラフトの「ボタンに軽い影は残してよい」は不採用——1箇所でも残すと「影は使ってよい」の前例になり漂流が再発する。押下フィードバックは transform と色で足りる。
- ドラフト提案の新罫線トークン `--hair-2`（16%）は不採用——既存 `--line-2`（22%）が同役割を担え、トークン追加は最小限の原則に反する。**強い区切り=`--line-2`（セクション外周）／弱い区切り=`--hair`（カード・行間）／最弱=余白のみ**の3段で統一。
- コーチ付き/自走の判別（旧 影差）: coach=`--surface` 面＋`--line-2` 罫線／self=`--bg` 面＋`--hair` 罫線、＋§3.3 の dot・ラベル。**グレースケール印刷でも罫線の濃度差とdot・ラベルで判別が残る**（面のクリーム差は印刷で消えるため冗長系を必ず併用）。
- 詳細オーバーレイのパネル: `--surface` 面＋`--line-2` 罫線＋`--scrim` 暗幕。パネルの浮きは幕のコントラストで担保（影は使わない）。
- 角丸は維持しつつ過大値を整理: 18〜22px→14px上限（外周）、カード10〜14px→10px、ピル `999px` はタブ・チップのみ維持。
- Hallmark 検査観点: 罫線は全周 `border`（**border-left/top 単独の色帯は引き続き禁止**）。`BLOCK_TINT` の種別色はテキスト色・dot にのみ使用（現状どおり）。

### 4.3 印刷適性（必須要件）

- 画面と印刷が同一構造（罫線ベース）になり、print 専用差し替えは「操作系・オーバーレイ非表示／タブ全展開／page-break」の3点だけに縮む。
- 背景は印刷時 `background:#fff` 既存規則を維持。面の濃淡が消える前提は罫線・dot・ラベルで冗長化済み。
- 検証規約: 印刷プレビュー（A4縦・背景グラフィックなし）で「日・週」の区切りが全て判別でき、1px罫線が出ること。

### 4.4 実装箇所

`ui/render-shared.mjs` `TOKENS`・`BASE_CSS` 全面（box-shadow 全置換・print 簡素化）、`ui/pattern-timeline.mjs` `PATTERN_CSS`（`.tlcard`・`.spine-*`・`.ev`・`.weekgrid`・`.wg-*`）、`ui/build.mjs` の index 用 `.pcard`。

## 5. 週「練習なし」表記と目盛り規則化（before→after）

### 5.1 「練習なし」表記の廃止

| 箇所 | before（実出力） | after |
|---|---|---|
| 畳みブレイク帯ラベル `.wg-breaklabel`（`weekLevel()` 内 `breakHtml`） | `午前〜午後（練習なし・11:43〜16:05）`（ピル・時刻レンジ付き） | **「…」（三点リーダー1字・10px mute・ピル背景なし）のみ**。ガターの時刻の飛び（12:00 の次が 16:00）と細い水平線（既存 `.wg-break::before/::after`）が省略を十分伝える。`aria-label="昼間の空き時間（省略）"` でアクセシビリティのみ担保 |
| 空曜日列 `.wg-noday` | `練習なし`（現データでは5日全てブロックありのため**非出現**） | `—`（emダッシュ1字）。非出現だが同型表記を残さない |
| 週レベル全体0件の注記 | `今週は練習なし。`（非出現） | `今週は予定が入っていません。`（ドラフト採用。データ0件の異常系**明示**は必要——ここは畳み表記ではなく状態説明なので省略記号にしない） |

### 5.2 時間軸目盛りの規則化（端数目盛り禁止・クランプ境界と目盛りの分離）

**現状**: `gutterTicks()` が「稼働帯の開始/終了時刻」＋「中間の毎時」を打つため、`11:43`・`16:05`・`17:45` の端数が目盛り・罫線に混入（実出力で確認）。

**決定**: 目盛り・罫線は**毎時00分のみ**。軸の表示範囲を外側の毎時に丸め、クランプ（イベントが乗る実稼働帯）はデータ導出のまま分離する。

| 項目 | before | after |
|---|---|---|
| 軸範囲 | `axisStart=09:00`・`axisEnd=17:45`（データ素値） | `axisStart = floor(データ最小開始, 60分)`＝09:00・`axisEnd = ceil(データ最大終了, 60分)`＝**18:00**（外側丸め。イベント配置は実時刻のまま） |
| 畳み区間 | `collapse = {11:43, 16:05}`（データ素値） | 検出は素値のまま、**表示境界を内側の毎時に丸める**: `collapseFrom = ceil(11:43→12:00)`・`collapseTo = floor(16:05→16:00)`。丸め後の幅が60分未満なら畳まない |
| 目盛り（`gutterTicks`） | `09:00,10:00,11:00,11:43,16:05,17:00,17:45` | **可視区間内の毎時のみ** → `09:00,10:00,11:00,12:00,16:00,17:00,18:00`（定間隔・定時刻。畳み前後の時刻の飛びは正） |
| 水平罫線（`rulePositions`） | 目盛りと同じ（端数位置にも線） | 毎時位置のみ（目盛りと常に一致） |
| 稼働帯の端数情報の行き先 | ガター目盛りに混入 | **列ヘッダ `.wg-dayhead .wt`（`16:05–17:45`）とイベント内開始時刻 `.evt`（`16:05`）が担う**（Googleカレンダーの作法: 軸は毎時固定・イベントが自分の実時刻を持つ） |

- ドラフトの変形案「軸クランプは端数のまま・目盛りラベルだけ毎時に絞る」は不採用——夕帯（16:05〜17:45）に打てる毎時が17:00の1本だけになり、目盛り間隔の規則性（等間隔に並ぶ毎時グリッド）が崩れる。外側丸めなら夕帯に16:00/17:00/18:00の3本が立ち、Googleカレンダーの「毎時スロット」の見えに揃う。丸めコストは軸延長 計37分×2px≒74px のみ。
- 丸めの算出はデータ導出（`floor/ceil` 演算。`12:00`/`16:00` をリテラルで書かない）。
- 日レベル中央スパインの時刻ピル（`16:05`・`16:55` 等）は**目盛りではなくイベント自身の開始時刻**なので端数のまま維持（丸めると実施時刻が嘘になる）。
- エッジケース: 全曜日連続帯→`collapse=null` で素の毎時軸／丸め後 `collapseFrom >= collapseTo`→畳まない（60分閾値に包含）。

### 5.3 実装箇所

`ui/pattern-timeline.mjs`: `buildWeekAxis()`（外側丸め・collapse境界丸め・60分閾値）、`gutterTicks()`（毎時のみに単純化）、`weekLevel()` の `rulePositions`（同）、`breakHtml`（「…」化）、`.wg-breaklabel`→`.wg-breakmark` CSS、`.wg-noday`・全0件文言。
