# 引継資料: 練習計画エンジン(コーチ関与3分類・土曜レクチャ・組違い)（2026-06-06）

## 概要

### このセッションで達成したこと
新チームのバスケ練習を「コーチの勘」から「変数(チーム事情・メンバー特徴・目標)→最適な週次練習メニューを自動生成する仕組み」に進めた。具体的には(1)出典つきドリル台帳211種の構築、(2)決定論エンジンMVP(週次プラン生成)、(3)コーチ関与3分類と男女合同・コーチ1人運用への対応(土曜まとめレクチャ・組違い)、を実装した。男女2グループをコーチ1人で見る運用の「組違い表」は暫定実装で、本来意図への作り直しが次の最優先。

### なぜやったか
最終ゴールは週次/日次練習プラン・選手個別アドバイス・試合戦術の3スキル。今回はその中核「変数→計画を計算するエンジン」と入力データ(ドリル台帳)を作るフェーズ。

## プロダクト方針(設計の前提・次セッションが守るべき軸)
- 決定論コードが計算・制約・配分・ゲートを担い、LLMは曖昧写像のみ(`engine/src/llm.js` はスタブ。自由記述コメント→既知カテゴリ写像のみが責務で、MVPは実呼び出しせずキーワード写像で代替・ネットワーク依存ゼロ)。LLMに時間配分や件数判定を丸投げしない。
- 保存層は抽象化(`engine/src/storage.js`)。今はローカルJSON、Firestoreは外販フェーズで差し替え(同じ3つのasync getter `getDrills`/`getConfig`/`getTeamInput` を team_id で実装するswap-in地点が定義済み・未実装)。エンジンのコアはUI/保存と分離。
- マルチテナント前提(team_idで引ける)・チーム哲学はconfigのパラメータ(ハードコードしない)。外販/配布を常に想定。
- MVP範囲: 1チーム・3指標(フリースロー成功率/試合ターンオーバー/ゴール下成功率)・週次プラン生成。サンプル入力は合成値(`engine/data/team-input.sample.json`、実選手データではない)。

### ハードゲート(`engine/src/gates.js`)
時間収まり/中学ゾーン禁止/年内セット禁止/成長期負荷上限(日・週の高強度本数・連続日回避)/守備・速攻の哲学フロア(best-effort=未達はthrowせず警告)/コーチ不在日の自走制限/アンダーフィル警告(著しく不足・空ブロックを警告)。

### 哲学フロア
守備(オールコートマンツー)・速攻(アーリー)を毎週最低分、コーチ在席日に必ず入れる(`config.philosophy_floors`。サンプルは各40分/週・在席日配置)。

### コーチ在席パラメータ
`config.schedule` の各日 `coach_present`(火金=在席/水木=不在、土=在席は仮置き・要確認)。不在日は自走系(ファンダ・シュート・ハンドリング・フィニッシュ・意思決定/ゲーム・コンディショニング)かつ習熟「反復/実戦化」に限定(`filter.js` の coach_absent_allow / isCoachAbsentEligible)。

### 成長期負荷モデル(`engine/src/loadModel.js`)
intensity_class(低/中/高=着地衝撃と全力度)で高強度の日次/週次上限・連続日回避を追跡。HIIT等の追込系は中に訂正済み、クールダウンは低強度ストレッチのみ(跳躍語を含む低強度ドリルもCDから除外)。

### コーチ関与3分類(`engine/src/filter.js` coachingMode)
self(自走)/practice(実践=対人・守備・ライブ・新規習得でない要監督)/lecture(レクチャ=新規習得かつ非対人)。判定はオーナー例と一致(マイカン=自走/1on1・対人=実践/フォームシューティング等=レクチャ/ビアーフィニッシュFIN-011=要コーチ override)。件数 **自走82/実践85/レクチャ44**(要コーチ計129)。レコードに `needs_coach` / `coaching_mode` の明示overrideがあれば派生より優先。

### 土曜まとめレクチャ(`engine/src/lecture.js`)
既習(introduced)リストに無いレクチャ型ドリルの導入を土曜(コーチ在席・最長枠)に男女合同で配置、introduced更新。導入後は平日に自走で反復(時系列)。生成計画の `plan.introduced` を次週の `config.introduced` に渡すと既習分は土曜レクチャから消える。

## 開発中に得た教訓
1. **テスト緑≠正しい出力**。何が起きたか=守備0配分・月〜金同一・HIITがクールダウン・組違いの矛盾、をテストは緑のまま通した。再発防止=ハードゲートや件数だけで合格にせず、メインAgentが本番経路の実出力(週次プラン全文)を毎回自分で読んで観点判定する。反映先候補=クロスセッション記憶。
2. **仕様を渡す側(PdM)の曖昧さがそのまま実装ズレになる**。何が起きたか=組違いを「同一メニュー・非担当は自走」と曖昧に指示し、結果「同じドリルを片方実践片方自走」という破綻実装になった。再発防止=「組違い=同時に別の活動」のような中核概念は実装前に1例で挙動を確定してから渡す。反映先候補=クロスセッション記憶。
3. **分類・強度ラベルは出典/定義を先に固める**。何が起きたか=大ジャンプ分類が出典なしの主観で誤り、HIITが強度低の誤ラベルでクールダウンに混入。再発防止=強度=着地衝撃と全力度、で定義し一次情報を当ててから分類する。反映先候補=クロスセッション記憶。

1. **男女組違いモデルの作り直し(最優先)**。
   現状(`engine/src/groups.js`)は「男女が同一ドリルを同時にやり、コーチが付く側=実践/付かない側=同じドリルを自走」とトグル表示する暫定実装で、(a)同じドリルが片方実践・片方自走で矛盾、(b)要コーチドリル(対人など)を非担当グループが無監督でやる破綻、がある。
   オーナー合意済みの正しいモデルへ作り直す: **「同時に"別の"活動をやる。男子がコーチ付きで実践ドリル→その間女子は別の自走ドリル→入れ替え。要コーチ(practice)ドリルは必ずコーチが付き、空いたグループは self(自走)ドリルで時間を埋める。両グループが最終的に同じメニューを一通りこなす」**。
   `groups.js` の割当ロジックを組み直し、不変条件「同時に2グループがpracticeにならない」「practiceドリルは必ずコーチ担当グループだけが実施」をテストで固定。実装後にメインが `node bin/plan.js` の組違い表を実物確認。
   - 参考: 現状の `groups.js` には不変条件チェッカー `findSimultaneousPractice` が既にあるが、これは「同一スロットで2グループがpractice」を検出するだけで、(b)の「非担当グループが要コーチドリルを自走」は検出しない。新モデルでは「片方がpracticeのスロットで、もう片方は同じ(要コーチ)ドリルではなく別のselfドリルを実施」を不変条件に追加する。
