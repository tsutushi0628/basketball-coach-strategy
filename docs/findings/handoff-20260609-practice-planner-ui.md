## 開発中に得た教訓
1. **複数パターン指示を1つに縮めない**: 何が起きたか=「何パターンか出して」に対し1案で提出し却下。再発防止=「複数/何パターン/案を並べて」は最低3案を実データで並列提出してから吟味に回す。反映先=Wasurenagusa/記憶。
2. **オーナーのズボラ表記を模倣しない（見栄え優先）**: 何が起きたか=実Docの`1715`無コロンを真似て見栄えを落とした。再発防止=配布物・UIの表示規約は見栄えの良い正規形（時刻はHH:MM・最小単位は人が読みやすい5分）を選ぶ。「実物に寄せる」は構造の話で、ズボラな表記まで写すことではない。反映先=記憶。
3. **「ちゃんとしたデータ」=意味的に正しい中身**: 何が起きたか=「データ接続」を配線の意味に矮小化し、ブロックにカテゴリ不適合ドリル（ゲームにマイカン）が混入したまま提出。再発防止=配布物に載せる前に各要素が業務的に妥当か（ドリルが正しいブロックにあるか）を主体が目視検証。データ品質はパイプラインの有無ではなく中身の妥当性。反映先=記憶。
4. **「実物を見たでしょ？」に推測で乗らない**: 何が起きたか=オーナーが見せたつもりの実Docがこちらに届かず特定に手間取った。再発防止=参照物が手元に無いと判明したら勝手に「あんな感じ」を作らず、Drive検索→特定できなければ実物（リンク/貼付/PDF）を1問で求める。反映先=記憶。
