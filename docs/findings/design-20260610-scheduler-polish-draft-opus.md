## 0. 専門観点の点検結果（着手前のゲート）

**判定: 同意して着手（ただし差し戻し1の遷移方式について代替案を1点、前提確認を1点添える）。**

Why/What/Done を行動観点で点検した結論は「5点とも採用UIの核を壊さずに実装できる」。ただし2点だけ先に確定させる:

- **代替案（差し戻し1の遷移方式）**: 「別ページ生成」と「ページ内詳細」の2択のうち、**ページ内オーバーレイ（モーダル相当の詳細レイヤー）を採る**。理由は §2.1 に詳述するが、行動観点の核は「コーチは練習中に体育館で片手スマホでタイムラインを上から追っている。ドリル名をタップ→詳細→戻る、を1タップずつで往復できることが Ability の要。別ページ遷移はブラウザの戻る／配布PDF（1ファイル前提）と衝突し、往復コストが上がる」。単一HTML自己完結（`build.mjs` が1ファイル出力・`clientScript` のタブ切替と同型）とも整合する。

- **前提確認（差し戻し1のデータ品質）**: ドリルの「実施方法」テキスト（`notes` フィールド）は214件すべてに存在するが、**132件にコーチ向けの出典補完メモ（「要出典補完」「出典実在を確認(直fetchは403…)」「指標化候補」等）が混在**している（§1.3）。これは配布画面にそのまま出すと「AIが作った調査メモ感」が出てオーナー酷評の温床になる。**本仕様は notes をそのまま全文表示せず、配布に適する部分だけを決定論ルールで切り出して表示する**方針を採る（§1.4）。この方針でよいか、もし「notes は触らず全文出す」ならその旨を差し戻してほしい（1問1答）。

それ以外（2フラット化・3タイポ・4週練習なし・5目盛り）は専門観点でも素直に採用UIの改善に乗る。異見なし。

## 1. データ棚卸し（メニュー詳細に出せる実在情報の一覧と欠損）

### 1.1 データの所在と流れ（確認済み）

```

   ↓ normalizeDrills（engine/src/normalize.js）：全記述フィールドを保持（court→enum化等の派生も追加）
   ↓ planWeek（engine/src/planWeek.js）：plan item は drill_id/name/minutes/category/intensity_class/needs_coach/coaching_mode/alternatives[{drill_id,name}] を持つ
   ↓ buildSession（ui/plan-data.mjs）：UI item は name/minutes/category/mode/video/alternatives(名前のみ) に絞る ← drill_id を捨てている（要修正・§6 T1）

```

**確認した事実（実データ実行で検証）**:
- カタログ全214件で記述フィールドはほぼ100%充足。ドリル名は214件すべて一意（名前での照合が安全）。

- 当月（8月）の計画に実際に登場するドリル名（本体＋alternatives＋WU components）は46種で、**46/46 がカタログに名前照合で解決**（未解決ゼロ）。

### 1.3 欠損・データ源に無い情報（でっち上げ禁止）

設計に出せない／無いものを明示する（オーナー差し戻しの「無いデータをでっち上げない」に対応）。

- **手順の番号付きステップ**: カタログに「①②③」の構造化手順は無い。`notes` は1〜数文の自由文。→ 詳細は「実施方法（自由文1段落）」として出す。番号付きステップ表示は作らない（捏造になる）。
- **図・コート図**: 静的画像のコート図は無い（`build.mjs` フッタも「コート図はドリルの動画リンクで代替中・手描き図の差し込み枠は次版」と明記）。→ 詳細に図枠は置かない。動画があるドリルは動画ボタンで代替。
- **動画はほぼ無い（24/214）**: 大多数のドリルに動画リンクが無い。→ 動画ボタンは「あるドリルだけ」出す。無いドリルで「動画なし」と冗長に書かない（さりげなく非表示）。
- **WU/CD の集約見出し（例: ダイナミックストレッチ）**: WU バンドは `ダイナミックストレッチ` を主見出しに micro動作（components）を畳んだ presentation。主見出し名 `ダイナミックストレッチ` はカタログにドリルとして実在（CND-001相当）し詳細を持つ。components（「肩回し」等）は micro 動作で個別ドリル詳細を持たない場合がある。→ **WU/CD バンドはバンド主見出し1件だけ詳細リンク可**、components は詳細リンクにしない（バンド詳細の本文に「内訳：…」として既存表示のまま列挙）。
- **選手個別データ**: 未接続・合成値（既存どおり）。詳細画面に選手別の数値は出さない。

### 1.4 `notes` の配布用クレンジング（決定論・LLM不使用）

`notes` は「実施方法」の主テキストだが、132/214 にコーチ向け出典補完メモが混じる。**配布画面に出すのは「やり方」の部分だけ**にするため、決定論ルールで末尾の出典・調査メモを落とす。

**切り出しルール（コード・正規表現で実装。`ui/plan-data.mjs` か詳細registry側）**:
1. `notes` を句点（`。`）で文に分割する。
2. 次の語を含む文を「出典・調査メモ」とみなして落とす（先頭一致でなく含有判定）: `要出典補完` ／ `出典` ／ `fetch` ／ `403` ／ `検索取得` ／ `検索で本文一致` ／ `サイト本文` ／ `ページ正式名` ／ `ページ正式` ／ `指標化候補` ／ `別エントリ参照` ／ `（直fetch`。
3. 残った文を `。` で再結合して「実施方法」本文にする。
4. 残りが空（全文が出典メモだった）になるドリルは、本文を出さず `sub_skill`＋`metric_meaning` だけで詳細を構成する（沈黙フォールバックにせず「実施方法はカタログに未記載」と1行明示する）。

- これは LLM 不使用の純粋な文字列処理（行動原則・LLM設計原則に整合）。判定語は固定リストで、入力に無い語を生成しない。
- 切り出しは「データ源に在るものだけを、配布に適する形で見せる」整形であり、情報の捏造ではない（捨てるのは内部メモ、残すのはやり方）。

## 2. 画面構成の決定（メニュー詳細の遷移方式・レイアウト・全状態）

### 2.1 遷移方式: ページ内オーバーレイ詳細（確定）

**決定**: タイムライン上のドリル（ドリル名）をタップすると、**同一ページ内のオーバーレイ詳細パネル**（画面手前に重なるシート）が開く。別ページHTMLは生成しない。

**現状→決定→理由**:
- **現状**: 日ビューの下に `<details class="menu-detail"><summary>共通メニュー…</summary>` の折りたたみがあり、開くと共通メニュー全段が縦に展開する。ドリル個別の詳細は無い。タイムラインのドリル名はただのテキスト（リンクでない）。
- **決定**: 折りたたみ「共通メニュー」を廃止し、タイムライン／週グリッド／2列スパインのドリル名を「タップ可能」にする。タップで `#drill-<id>` の詳細オーバーレイを開く。閉じるは ✕／背景タップ／Esc。
- **理由（行動観点）**: コーチは体育館で片手スマホ、タイムラインを上から追う。ドリル名タップ→実施方法を確認→閉じてタイムラインの続きへ、を最小タップで往復したい。別ページ遷移だとブラウザ戻る依存＋配布PDF（単一ファイル・印刷）と衝突する。オーバーレイなら `clientScript` のタブ切替と同型のDOM表示切替（hidden属性トグル）で済み、印刷時はオーバーレイを `@media print` で非表示にして本体だけ印刷できる。

**静的HTML制約下の実装方式**: `build.mjs` はビルド時に全ドリルの詳細パネルHTMLを**1ページ内にまとめて生成**（全件 hidden）。`clientScript` に「ドリル名クリック→対応する詳細パネルを表示／閉じる」ハンドラを追加（既存のタブ切替と同じ DOM hidden トグル方式）。追加スクリプトは10〜20行程度。サーバ不要・単一HTML自己完結を維持。

### 2.2 詳細パネルのレイアウト（タイポ階層は §3 に従う）

詳細オーバーレイ1枚の構成（上から）。全要素を §3 のタイポ段に割り当てる。

```
┌─ オーバーレイ（背景は半透明の薄暗幕・パネルは --surface 面）────────┐
│  [✕ 閉じる]                                              （右上・小）│
│                                                                      │
│  ●コーチ付き / 自走                          （モードタグ・小）       │
│  ドリル名                                    （H2相当・大・主役）     │
│  何の練習か（sub_skill）                     （サブ見出し・mute）     │
│                                                                      │
│  ── 狙い ──────────────────────────────────  （ラベル・小・mute）   │
│  metric_meaning（何を伸ばす／測るか）        （本文）                │
│                                                                      │
│  ── 実施方法 ──────────────────────────────  （ラベル・小・mute）   │
│  notes（クレンジング済み・無ければ「カタログに未記載」）（本文）     │
│                                                                      │
│  ── メタ（横並びチップ）─────────────────────────────────────────── │
│  [35分] [強度:中] [コート:半面] [ボール:1球] [人数:3人交代] [反復]   │
│  （注意があれば）注意：load_notes              （小・mute）          │
│                                                                      │
│  [▶ 動画を見る]（video_url がある時だけ）      （ボタン・既存.vid流用）│
│                                                                      │
│  ── いずれか（代替候補）──  ※本体ドリルのみ・WU/CDは出さない        │
│  ・代替ドリル名1（タップで詳細へ）            （リスト・各々リンク）   │
│  ・代替ドリル名2                                                     │
│                                                                      │
│  出典：source_name（source_url があればリンク） （末尾・最小・mute） │
└──────────────────────────────────────────────────────────────────┘
```

- **トンマナ**: パネルは `--surface` 面、幕は半透明の暗幕（`rgba(42,32,26,.32)` 程度・既存トークン外だが幕用に1値追加可・§3.5）。角丸は既存（18〜22px）。**シャドウは §4 のフラット方針に従い、パネルの浮きは細い罫線＋幕のコントラストで出し、ぼかし影を主強調に使わない**。
- **メタチップ**: 既存の `.dh-court`（inset ピル）と同系。border-left 色帯は使わない。強度は文字（「強度:中」）で出し、色帯にしない。
- **動画ボタン**: 既存 `videoLink`／`.vid`（SVG再生アイコン・emoji不使用）を流用。
- **代替候補**: 各候補もドリルなので、候補名タップで**その候補の詳細オーバーレイに差し替え**（同じオーバーレイ機構を再利用）。

### 2.3 全状態パターン（詳細情報の濃淡で崩れない設計）

| 状態 | 条件 | 詳細パネルの見せ方 |
|---|---|---|
| フル情報 | notes（クレンジング後）あり・動画あり・代替あり | 全セクション表示。 |
| 動画なし（多数派・190/214想定） | `video_url` 無し | 動画ボタンを出さない（「動画なし」と書かない）。 |
| 実施方法が内部メモのみ | クレンジング後 notes が空 | 「実施方法」見出しの下に「このドリルの手順はカタログに未記載（狙いとメタを参照）」と1行（沈黙フォールバック禁止・明示）。 |
| 代替なし | alternatives 0件（WU/CD バンド・単独ドリル） | 「いずれか」セクションを出さない。 |
| WU/CD バンド主見出し | `isBundle` のバンド（ダイナミックストレッチ等） | 詳細を出すが「いずれか」は無し。本文に既存の components 内訳（「内訳：肩回し／…」）を併記。 |
| 出典なし | `source_url`／`source_name` 無し | 出典行を出さない。 |
| コーチ不在日の自走ドリル | `mode==='self'` | モードタグ「自走」。詳細の中身は同じ（ドリル自体は不変）。 |

- **タップ対象が無いドリル名は無い**（46/46 解決済み）。万一 registry に未解決のドリル名が来たら、その名前はリンク化せず素テキストのまま出す（黙ってリンク切れにしない・クリックで空パネルを出さない）。

### 2.4 折りたたみ「共通メニュー」の廃止（差し戻し1の本体）

- `dayTimeline()` の `<details class="menu-detail">…<summary>共通メニュー（男女同一の内容）</summary>${menuTimeline(pd)}</details>`（`pattern-timeline.mjs:171`）を**廃止**。
- rotation 日は中央スパイン3列のみ、非rotation 日は `menuTimeline(pd)` を `<details>` でなく**直接展開**（折りたたみ自体をやめる）。
- ドリルの実施方法は「折りたたみで全段一覧」ではなく「各ドリル名タップ→そのドリルの詳細オーバーレイ」に置き換わる。これがオーナーの「折りたたみ開示は廃止・クリックで詳細に飛ぶ」の意図。
- `@media print` では全ドリルの詳細を畳んだままにし（オーバーレイは印刷非表示）、印刷はタイムライン本体だけ出す。実施方法を印刷に含めたい需要が将来出たら別タスク（本スコープは画面遷移の確立まで）。

### 3.3 「コーチ付き」ラベル・カード3行の凝縮（差し戻し3の具体）

**現状の問題（実画面確認）**: rotation スパインのカード（`.spine-side`）と週グリッドのカードが「●コーチ付き（オレンジpill・大）／ドリル名／分数」で3行を消費し、オレンジpillのコーチ付きタグがドリル名と同等以上に目立つ。オーナー指摘「コーチ付きラベルが大きすぎ・3行でスペース無駄」。

**決定（凝縮ルール）**:
1. **コーチ付き/自走は「タグpill」をやめ、ドリル名の行頭に小さな点リード（dot）＋極小ラベルで表す**。

2. **カードを2行に凝縮**: 1行目＝「[モードdot] ドリル名（D3）  …  分数（D6・右寄せ）」、2行目＝「ブロック名（D5・mute）」。分数を名前と同じ行の右端に寄せ、3行目を消す。

3. **詳細オーバーレイのモードタグ**も同じ dot＋極小ラベル方式で統一（pillの大きいコーチ付きタグを画面から一掃）。
4. `render-shared.mjs` の `modeTag()`（現 `.tag-coach` オレンジpill）は、**dot＋極小ラベル方式の新ヘルパー `modeMark()` に置換**（既存 `modeTag` を呼ぶ箇所を差し替え。`.tag`/`.tag-coach`/`.tag-self`/`.tag-lec` CSSは詳細・凡例用に最小残すか削除）。

これにより「重要（ドリル名）は大きく＝D3、重要でない（モード・分数）は小さく＝D6」が規則として成立する（差し戻し3の核）。

## 4. フラット化の具体（廃止するシャドウ・代替の区切り・印刷適性）

### 4.1 現状の問題（実画面・印刷観点で確認）

`render-shared.mjs` の `--shadow`／`--shadow-soft`／`--inset`（ニューモーフィズム）が全カードに乗っており、実画面で各カードの周囲に柔らかいぼかしハロが出る。オーナー指摘「変なシャドウわかりづらいし印刷しづらい」。印刷時、ぼかし影はグレーの滲みになって罫線として機能せず、インク無駄＋境界が曖昧になる。

### 4.2 決定: 影依存をやめ、罫線＋面の濃淡で区切る（フラット気味）

**原則**: 区切りは「①薄い面の濃淡（`--bg`／`--surface` の差）＋②細い罫線（`--hair`／`--hair-2`）」で出す。**ぼかし影（box-shadow のブラー）は主たる区切り手段から外す**。完全な影ゼロにはしない（ボタン押下フィードバック等の軽い影は可）が、カードの「浮き」を影で作るのをやめる。

| 現状（影で浮かせる） | 決定（フラット） |
|---|---|
| `.dayhead`／`.goals`／`.monthcard`／`.note` 等が `--shadow`／`--shadow-soft` で浮く | 面は `--surface`、外周は `--hair-2` の1px罫線。box-shadow は除去（または極薄の1段だけ残す）。 |
| カード内の小ピル `--inset`（彫り込み影） | inset を `--bg` 面＋`--hair` 1px罫線に置換。彫り込み感をやめフラットなチップに。 |
| `.ev.coach`／`.spine-coach` が `--shadow-soft` で浮き、`.self` が `--inset` で凹む | コーチ付き＝`--surface` 面＋`--hair-2` 罫線、自走＝`--bg` 面（罫線なしor `--hair`）。**面の明暗差**でコーチ付き/自走を区別（影でなく面で区別）。 |
| 週グリッドのイベント影 | 同上。`--bg` 面＋`--hair` 罫線。コーチ付きのみ `--surface` 面で一段明るく。 |
| 詳細オーバーレイのパネル | `--surface` 面＋`--hair-2` 罫線＋`--scrim` 幕。パネル自体の落ち影は最小（幕のコントラストで浮きを担保）。 |

- **コーチ付き/自走の区別は「面の明暗」で担保**（コーチ付き＝明るい `--surface`、自走＝地の `--bg`）。これは影に依存しないので印刷でも残る。
- ボタン（`.btn`／`.lvtab`／`.daytab`）の押下/ホバーの軽い影は操作フィードバックなので残してよい（区切りの主手段ではない）。

### 4.3 印刷適性（必須要件）

- `@media print` で box-shadow を全廃し、カードは `--hair-2`（または既存 `--line-2`）の1px罫線に置換（既存 print ルール `box-shadow:none;border:1px solid var(--line-2)` の方針を全カードに徹底）。
- 面の濃淡（`--bg`／`--surface`）は印刷で薄いグレー差として残るが、罫線が主境界になるので潰れない。
- 詳細オーバーレイ・幕（`--scrim`）・モードdot は印刷で意味を持たないので、オーバーレイは `@media print { display:none }`。本体タイムラインだけ印刷。
- A4で罫線が細すぎないこと（1px罫線が出るか）を frontend が印刷プレビューで確認。

## 5. 週「練習なし」と目盛り規則化の具体（before→after）

### 5.1 週「練習なし」表記の廃止（差し戻し4）

**現状（実画面）**: 畳みブレイク帯に「午前〜午後（練習なし・11:43〜16:05）」（`pattern-timeline.mjs:349`）。さらに練習0件曜日の列に「練習なし」（`.wg-noday`・`:357`）。

**決定**: 「練習なし」の文言を撤去し、省略記号的表現にする。

| 箇所 | before | after |
|---|---|---|
| 畳みブレイク帯ラベル（`:349` `wg-breaklabel`） | 「午前〜午後（練習なし・11:43〜16:05）」 | 「⋯（あいだの時間）」程度の省略表現。時刻レンジは出すなら畳み境界の時刻のみ（§5.2の規則目盛りに従う）。文言で「練習なし」と言わず、畳まれている事実だけを `⋯` と薄い面で示す。 |
| 練習0件曜日列（`:357` `wg-noday`） | 「練習なし」 | その曜日は列ヘッダの開始終了を空にし、列内は薄い地のまま（`⋯` か空）。「練習なし」と書かない。※当月の5練習日は全て練習ありなので通常は出ないが、データ次第で出る分岐は残す。 |

- 「⋯」は装飾でなく省略記号（ellipsis）。emoji・特殊アイコンは使わない。薄面（`--bg`）＋mute の `⋯` 文字だけ。
- 畳みブレイク帯は §4 のフラット方針に従い、ぼかし影でなく上下の `--hair` 罫線＋薄面で「畳まれている帯」を示す。

### 5.2 時間軸目盛りの規則化（差し戻し5）

**現状（実画面・最重要バグ）**: ガター目盛りが `09:00 / 10:00 / 11:00 / 11:43 / 16:05 / 17:00 / 17:45` のように、**規則的な定時（毎正時）と稼働帯の端数境界（11:43・16:05・17:45）が混在**。オーナー指摘「11:43 のような端数目盛り禁止」。原因は `gutterTicks()`（`:264`）が「各稼働区画の from/to（＝端数時刻）」も目盛りに打っているため。

**決定: 目盛りは定間隔の定時刻のみ（毎正時）。稼働帯のクランプ境界（端数）は目盛りから分離する。**

```
現状の gutterTicks のロジック:
  各稼働区画について {
    push(区画開始時刻)          ← 09:00/16:05 等（端数が混入）
    push(毎正時 nextHour..to)   ← 10:00/11:00/17:00（これは規則的でOK）
    push(区画終了時刻)          ← 11:43/17:45 等（端数が混入）★これが禁止対象
  }

決定後の gutterTicks のロジック:
  各稼働区画 [from,to] について {
    毎正時のみを打つ: ceil(from/60)*60 から floor(to/60)*60 まで 60分刻みで push
  }
  ※区画の from/to（端数）は目盛りに打たない。
  ※稼働帯のクランプ境界（端数）は「列ヘッダの開始〜終了」で別に明示する（既に .wt にある）。
  ※軸の物理的なクランプ（axisStart/axisEnd が端数でも）は配置計算には使うが、目盛りラベルは毎正時だけ表示。
```

| before（ガター目盛り） | after |
|---|---|
| 09:00 / 10:00 / 11:00 / **11:43** / **16:05** / 17:00 / **17:45** | 09:00 / 10:00 / 11:00 ／（畳み）／ 17:00 ※毎正時だけ |

- **稼働帯の端数（09:00開始・11:43終了・16:05開始・17:45終了）は「列ヘッダの 開始〜終了」（`.wt`）で見せる**（既存実装にある「火 16:05–17:45」「土 09:00–11:43」）。これでクランプ境界の情報は失われない。目盛りと境界表示を分離するのが決定の核。
- 09:00 ちょうどや 17:00 のような「区画内に入る毎正時」は打つ（09:00 は土の区画 from=540 がちょうど毎正時なので打たれる・OK）。16:05 のような毎正時でない区画境界は打たない。
- 水平罫線（`.wg-rule`・`rulePositions`・`:330`）も同じ規則に揃える: 端数境界の罫線をやめ、毎正時だけに罫線を引く（目盛りと罫線の位置を一致させる）。畳みブレイクの上下罫線は別（畳みの境界として残す）。
- **実データ追従は維持**: 毎正時の算出は `axisStart/axisEnd`・`used` 区画からデータ導出（11:00 や 17:00 をリテラルで書かない・`ceil(from/60)*60` で算出）。畳み・クランプのデータ導出ロジック（`buildWeekAxis`）は不変、目盛りラベルの選び方だけ毎正時に絞る。
